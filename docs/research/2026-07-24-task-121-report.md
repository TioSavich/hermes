# Task 121 report: the field-context hang, named and bounded

## Result

The field-context band freeze is not slowness misread. A class of mid-grade
lessons spends tens of seconds inside native Prolog builtins that never poll for
the asynchronous alarm `call_with_time_limit/2` installs, so the advertised
120-second per-lesson limit cannot preempt them. The limit's scope is correct —
it wraps the whole per-lesson goal — but its mechanism cannot interrupt a foreign
or builtin section. Under shared tabling with eight workers a wedged lesson holds
its thread and blocks its peers, and the band reports no further progress while
holding its nodes: exactly the 06:15–06:30 freeze the cluster logs recorded.

The fix moves the wall outside Prolog. Python now watches the completed-lesson
cadence and, when a batch produces nothing within `--lesson-timeout` seconds,
terminates it, resumes from the atomic partials, and — once no concurrency makes
progress — isolates one lesson per worker so the next stall names an exact
culprit, which it records as an explicit error and steps past. A single
pathological lesson can slow a band but can no longer stall it.

## The mechanism, proved not conjectured

All timings are local (macOS, SWI-Prolog 9.2.9), single-threaded unless noted,
each reproduction under an external `gtimeout` kill so nothing ran unbounded and
the full 1,317-lesson rebuild was never started.

1. **The alarm itself works.** `call_with_time_limit(2, (between(1,inf,_),fail))`
   throws `time_limit_exceeded` at 2.002 s — in plain `swipl`, with
   `hermes_worker.pl` loaded, and after `load_runtime`. The guard is not broken
   in the batch environment.

2. **The guard does not fire on the real work.**
   `call_with_time_limit(5, monitoring_chart_export('IM-G4-U3-L12', _))` runs to
   completion at **37.7 s** with no timeout raised (untabled). Tightening the
   limit does not help: `call_with_time_limit(5, lesson_strategy('IM-G4-U3-L12',
   …))` returns normally at **35.2 s**. The alarm is registered but cannot be
   delivered until the builtin returns, by which point the goal has already
   finished.

3. **Where the time goes.** Splitting `monitoring_chart_export/2` for
   `IM-G4-U3-L12`: `lesson_standards` 0.9 s, `lesson_misconception` 1.4 s,
   `lesson_pml_facts` 0.05 s, cluster `findall` 0.05 s — and `lesson_strategy/4`
   **35.1 s**. Inside `lesson_strategy/4` the cost is the first, uncached
   evaluation of `lesson_topics/2`, reached through the geometry-fallback clause
   of `lesson_strategy_candidate/4`:
   `geometry_concept_fallback_allowed/1` measures **35.1 s and returns false**.
   `lesson_topics/2` runs `topic_from_text/2` — chained `sub_atom/5` substring
   scans plus `string_lower/2`/`atom_string/2` — over the `im_lesson` standard
   anchors in `curriculum/im/lesson_monitoring.pl`. These are C builtins; they do
   not check for pending signals, so the alarm cannot land inside them.

4. **Why the whole band, not one lesson.** The cost is per lesson, not a
   one-time process warmup: in one process `lesson_topics` measured 28.5 s for
   `IM-GK-U1-L1` and 35.1 s for `IM-G4-U3-L12`. Each concept carries ~299–598
   anchor rows and ~120–160 kB of statement text. The result caches in
   `cached_lesson_topics/2` after the first touch (a second evaluation is
   0.000 s), so a single process pays it once per distinct concept. With eight
   shared-tabling workers the tabled path is slower still and peers block in
   condition-variable waits on a peer-held shared table — also a non-interruptible
   section — so no worker can process its own alarm and the band's log and
   partials stall together.

Answer to the brief's precise question: the 120-second limit did not fire because
`call_with_time_limit/2` is an alarm delivered only at Prolog VM signal-check
points, and the pathological lessons spend their time in native builtins
(`sub_atom/5` and friends inside `lesson_topics/2`/`topic_from_text/2`) that never
reach such a point until they complete. This is the brief's candidate
"call_with_time_limit vs a blocking/foreign section alarms cannot preempt," now
measured rather than assumed.

## Stuck lesson id(s)

`IM-G4-U3-L12` is the demonstrated culprit and matches the cluster evidence
exactly: band 1's last checkpoints were `IM-G4-U3-L1`, `-L10`, `-L11`, i.e. the
workers had reached the `IM-G4-U3` neighbourhood and wedged. The pathology is not
unique to it — every mid-grade lesson pays the same ~30 s uncached `lesson_topics`
scan (band 2's frozen frontier was the parallel `IM-G7-U4` neighbourhood) — so
the fix is not lesson-specific. `IM-G4-U3-L12` is the reproduction handle.

## The fix (scripts/research/build_field_context_cache.py)

The in-Prolog 120-second limit is left in place as a best-effort inner bound (it
still catches lessons whose time is in interruptible Prolog search, a different
class). The guaranteed backstop is external:

- `compute_contexts` gains a watchdog thread. It tracks the time since the last
  completed lesson; if that exceeds `--lesson-timeout`, it kills the SWI process
  and raises `StallTimeout`. Every lesson streamed before the kill is already
  checkpointed atomically, so nothing computed is lost.
- `collect_partials` becomes a resilient driver. On a stall it resumes from the
  partials on disk. If a full-concurrency batch made progress before stalling, it
  keeps going at full speed. If it made no progress, it drops to one lesson per
  worker; a single-lesson batch that stalls is unambiguously attributable, and
  that lesson is written as an explicit error partial and stepped past. The loop
  makes monotonic progress: each stuck lesson costs at most one full-batch window
  plus one isolation window, then is recorded and excluded.
- `write_error_partial` reuses the unchanged `serialized_partial`, so a recorded
  stall is byte-identical in shape to a Prolog-emitted error partial:
  `{"lesson_code": …, "context": {"error": …}}`. The recorded-failure-not-omission
  contract holds, and `--retry-errors` can recompute these later.
- `serialized_partial`/`read_partial`/`serialized_artifact`/`write_atomic` are
  untouched, so the 797 partials already on the cluster remain valid input and the
  merge stays byte-stable.
- New flag `--lesson-timeout` (seconds; default 600; 0 disables). `job.slurm`
  passes it through `FIELD_CONTEXT_LESSON_TIMEOUT` (default 600), keeping the
  nounset-safe env block and pre-created directories. The `--resume`/`--collect`/
  `--merge` invocation surface is otherwise unchanged.

600 seconds is a deliberately loose wall: a legitimate lesson (~30–90 s) never
trips it; a true wedge (zero completions for ten minutes across a whole batch)
does. The isolation pass then narrows to the exact lesson.

## Limits of this diagnosis

The local reproduction is single-node and single- or few-threaded; the precise
eight-worker shared-tabling interaction on Big Red (leader/waiter blocking versus
GC pressure versus swap) was inferred from the measured non-interruptibility plus
the cluster's frozen-frontier logs, not re-run at cluster scale. The external
wall does not depend on which of those compounds the freeze: it bounds forward
progress from outside the Prolog process regardless. The deeper performance
question — that `lesson_topics/2` costs ~30 s per concept at all — is real but
out of this task's fence (`curriculum/im/lesson_monitoring.pl` is read-only here);
it is a strong candidate for a later build-time memoization pass and would cut the
whole rebuild by roughly its dominant term.
