The new intake keeps the useful provenance and separate-description-pass
shapes. Corpus scale required these changes:

- Atomic Markdown and JSON checkpoints replace a shared append-only JSONL
  during conversion. Twelve array tasks can write separate journal
  directories without contending for one file.
- A parent watchdog runs one subprocess per PDF, applies a 1,800-second
  default external wall, records `*.error.json`, and continues.
- Resume validates source name, byte size, and nanosecond mtime before
  skipping a completed pair.
- A deterministic collector builds the shared JSONL index after conversion.
- Full Markdown is retained, including accurate table structure and
  CodeFormulaV2 equation enrichment.
- Figure description remains a second pass. Sidecars contain stable page and
  bounding-box locators, and the description-manifest mode emits
  Granite-style requests without calling a model.
- Bibliography matching reads both BibTeX files and uses file path, key and
  filename, DOI, title, and author evidence. Scores below 0.72 remain
  `unkeyed`.

## Implemented files

### Local driver

`scripts/research/literature_intake.py` supports:

- `--pdf` for one PDF;
- `--journal-dir` for a checkpointed journal batch;
- `--smoke` for the three small corpus papers and the first 15 pages of the
  Van de Walle, Karp, and Bay-Williams book;
- `--enumerate` for the live per-journal counts; and
- `--description-manifest` for the deferred figure-description queue.

Docling 2.112.0 runs with:

- RapidOCR through ONNX Runtime;
- accurate table structure;
- CodeFormulaV2 through the CPU Transformers engine; and
- no first-pass figure-description model call.

The explicit CPU engines avoid platform-dependent MLX and Apple Vision
selection in a headless macOS sandbox. On this machine the installed Docling
extensions are arm64, while the default terminal Python runs x86_64. The local
smoke therefore used `arch -arm64 /usr/local/bin/python3`. The driver preserves
that architecture when it starts macOS workers. Big Red uses its ordinary
Linux `python3`.

### Big Red array and collection

`scripts/bigred/literature_docling/job.slurm` maps array indices 0 through 11
to the 12 journal directories. It includes:

- `REPLACE_WITH_BIG_RED_ACCOUNT`;
- a nounset-safe module and Conda block;
- an explicit pre-created scheduler log path;
- one shared, resumable output root;
- one per-PDF external watchdog; and
- unbuffered progress from the Python driver.

`scripts/bigred/literature_docling/collect_merge.py` writes:

- `corpus_index.jsonl`;
- `corpus_failures.jsonl`; and
- `corpus_manifest.json`.

The helper sorts by journal and source filename and writes no timestamps.
`scripts/bigred/literature_docling/README.md` records staging, submission,
resume, collection, merge, and second-pass figure-request commands. The
controller must create
`scripts/bigred/literature_docling/logs` before `sbatch`, because Slurm opens
the scheduler output before running the script body.
