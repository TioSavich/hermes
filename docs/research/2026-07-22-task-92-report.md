# Task 92: Hermes rendered as an MCP server

## Result

Hermes now has a dependency-free, newline-delimited JSON-RPC 2.0 MCP server at `hermes/mcp/server.py`. It lazily starts one `PersistentPrologWorker` on the first tool call and closes it when stdin reaches EOF. It implements initialization, `notifications/initialized`, `tools/list`, and `tools/call`; malformed calls are JSON-RPC error objects rather than process failures.

The default `core` mode exposes ten interpreter-oriented tools. The `registry` mode reads the live capability registry and exposes all 242 current operations. The registry has operation names and parameter names, but not types, requiredness, or default values. Its generated schemas therefore declare unknown parameter types as strings and state that limitation.

`resonance_neighbors` never calls an embedding endpoint. It uses the selected misconception's stored vector from `data/research/misconception_embeddings.*` and compares that row with other stored rows.

## Findings: bridge defects and proposed fixes

1. **Registry-scale tool overload.** The live registry provides 242 tools, too many for a practical interpreter surface. **Fix:** retain the core mode as the default; add role- or task-specific curated bundles before considering registry mode for an interactive interpreter.

2. **Schema poverty.** The registry reports only parameter names. It cannot distinguish strings from JSON objects or lists, nor required inputs from defaults. **Fix:** extend registry extraction from `dispatch_spec/4` so it carries JSON types, requiredness, defaults, and examples.

3. **Worker latency is surfaced without guidance.** A first monitoring-chart attempt exceeded the worker's inherited 20-second request deadline. The MCP bridge now gives its one local worker a bounded 120-second request window, but a client still receives no progress information. **Fix:** add progress notifications or an explicit long-running-job surface for chart exports.

4. **Raw curriculum charts are too large for a tool reply.** A successful deformation chart carries many deeply nested scenes and frames. The payload is useful to a renderer but awkward for a general interpreter. **Fix:** offer a summary endpoint with identifiers and coverage first, followed by a targeted detail tool for one fraction, host, or deformation.

5. **Worker errors lose structured cause.** The worker returns a typed error, but the bridge currently exposes only its message in a generic `-32000` JSON-RPC error. **Fix:** preserve the worker error type in JSON-RPC error data and distinguish malformed input, absent coverage, and internal worker failure.

6. **Math checking is indirect.** The worker has no direct `check_math_claim` dispatch operation. The core tool uses `ground`, which presently parses fraction equivalences only. **Fix:** add a direct typed math-claim dispatch operation and publish its accepted claim grammar in the schema.

7. **Stored-vector resonance is name-led, not semantic query retrieval.** Offline operation requires selecting an existing stored misconception name; that is intentionally not a network embedding query, but it is not a free-text neighbor search. **Fix:** add an explicit row-picker/search tool or allow a caller-supplied vector with dimension validation, while keeping network embedding separate.
